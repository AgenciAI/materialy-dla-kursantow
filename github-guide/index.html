Przewodnik GitHub na macOS

Git & GitHub
od zera do backupu

Kompletna instrukcja instalacji i pracy z Git oraz GitHub CLI na macOS. Komendy po kolei, z opisami po polsku — dla Terminala i sesji Claude Code.

System: macOS
Menedżer pakietów: Homebrew
GitHub CLI: gh
Przygotowane przez: Agenci AI
1
Instalacja Homebrew
Homebrew to menedżer pakietów na macOS — potrzebny do instalacji Git i GitHub CLI. Jeśli go masz, pomiń ten krok.
Sprawdź czy Homebrew jest już zainstalowany
Jeśli polecenie zwróci wersję — Homebrew jest gotowy, przejdź do sekcji 2.
brew --version
# Homebrew 4.x.x  ← jeśli widzisz wersję, masz Homebrew
Zainstaluj Homebrew (jeśli nie masz)
Skopiuj i wklej całe polecenie do Terminala, naciśnij Enter. Instalacja trwa kilka minut — Terminal będzie pytał o hasło systemowe.
/bin/bash -c "$(curl -fsSL https://raw.githubusercontent.com/Homebrew/install/HEAD/install.sh)"
Dodaj Homebrew do PATH (tylko Apple Silicon M1/M2/M3/M4)
Na MacBookach z chipem Apple po instalacji musisz jednorazowo dodać Homebrew do ścieżki systemowej.
# Uruchom te dwa polecenia po kolei:
echo 'eval "$(/opt/homebrew/bin/brew shellenv)"' >> ~/.zprofile
eval "$(/opt/homebrew/bin/brew shellenv)"
✓
Weryfikacja: po dodaniu do PATH uruchom brew --version — powinnaś zobaczyć numer wersji.
2
Instalacja Git
Git to system kontroli wersji — śledzi zmiany w plikach i pozwala cofać się do wcześniejszych wersji kodu.
Sprawdź czy Git jest już zainstalowany
git --version
# git version 2.x.x  ← jeśli widzisz wersję, Git jest gotowy
Zainstaluj Git przez Homebrew
brew install git
Zweryfikuj instalację
git --version
3
Instalacja GitHub CLI (gh)
GitHub CLI to narzędzie gh — pozwala tworzyć repozytoria, zarządzać nimi i logować się do GitHub bezpośrednio z Terminala.
Zainstaluj GitHub CLI
brew install gh
Zweryfikuj instalację
gh --version
# gh version 2.x.x  ← poprawna instalacja
ℹ
GitHub CLI (gh) jest oddzielnym narzędziem od Git. Git zarządza plikami lokalnie, gh komunikuje się z serwerem GitHub przez API.
4
Konfiguracja tożsamości Git
Git musi wiedzieć kto robi zmiany — ta informacja trafia do każdego commita. Rób to jednorazowo po instalacji.
Ustaw imię i e-mail (zastąp swoimi danymi)
Użyj tego samego e-maila co na koncie GitHub.
git config --global user.name "Twoje Imię i Nazwisko"
git config --global user.email "twoj@email.com"
Ustaw domyślną gałąź na "main"
Nowoczesny standard — zamiast starej nazwy "master".
git config --global init.defaultBranch main
Sprawdź zapisaną konfigurację
git config --global --list
5
Logowanie do GitHub
Jednorazowe uwierzytelnienie GitHub CLI — po tym nie musisz wpisywać hasła przy każdym push.
Zaloguj się do GitHub przez CLI
Po uruchomieniu polecenia Terminal przeprowadzi Cię krok po kroku przez logowanie. Wybierz: GitHub.com → HTTPS → Login with a web browser.
gh auth login
Sprawdź status logowania
gh auth status
# ✓ Logged in to github.com account twoje-konto (keyring)
✓
Dane logowania są przechowywane w macOS Keychain — bezpiecznie, bez pliku z hasłem na dysku.
6
Inicjalizacja nowego projektu lokalnie
Jak zacząć śledzić zmiany w istniejącym folderze z projektem.
Przejdź do folderu projektu
cd "/ścieżka/do/twojego/projektu"
# Przykład:
cd "/Users/samil/Documents/MojProjekt"
Zainicjuj repozytorium Git
Tworzy ukryty folder .git/ — od tej chwili Git śledzi zmiany w tym folderze.
git init
Stwórz plik .gitignore (co ma być pomijane)
Wpisz nazwy folderów i plików które NIE mają trafiać do GitHub — dane, hasła, paczki npm.
# Przykładowy .gitignore dla projektu Node.js:
node_modules/
.env
data/
*.log
.DS_Store
Dodaj wszystkie pliki do pierwszego commita
git add .
git commit -m "Initial commit — pierwsza wersja projektu"
⚠
Ważne: przed git add . upewnij się że masz .gitignore z plikiem .env — nigdy nie wrzucaj kluczy API na GitHub.
7
Tworzenie repozytorium na GitHub i wysyłanie kodu
Jak stworzyć prywatne repozytorium na GitHub i wgrać do niego lokalny projekt jednym poleceniem.
Utwórz prywatne repo i wypchnij kod (zalecane)
Jedno polecenie tworzy repo na GitHub, łączy je z lokalnym projektem i wysyła kod. Uruchom z folderu projektu.
gh repo create nazwa-projektu --private --source=. --remote=origin --push
Z opisem repozytorium
gh repo create nazwa-projektu \
  --private \
  --description "Krótki opis projektu" \
  --source=. --remote=origin --push
Alternatywnie: ręczne podłączenie istniejącego repo
Gdy repo już istnieje na GitHub, a chcesz połączyć je z lokalnym projektem.
git remote add origin https://github.com/twoj-login/nazwa-projektu.git
git branch -M main
git push -u origin main
Sprawdź adres podłączonego repo
git remote -v
# origin  https://github.com/twoj-login/projekt.git (fetch)
# origin  https://github.com/twoj-login/projekt.git (push)
8
Codzienna praca — zapisywanie i wysyłanie zmian
Standardowy cykl pracy z Git: sprawdź co się zmieniło → dodaj → zapisz commit → wyślij na GitHub.
Sprawdź stan plików
Pokazuje które pliki zostały zmienione, dodane lub usunięte od ostatniego commita.
git status
Podgląd konkretnych zmian w plikach
git diff                    # wszystkie zmiany
git diff server.js          # zmiany w konkretnym pliku
Dodaj zmiany do staging (przygotowanie do commita)
git add .                   # wszystkie zmienione pliki
git add server.js src/db.js  # konkretne pliki
git add public/              # cały folder
Zapisz commit z opisem
Opis powinien mówić CO i PO CO zostało zmienione — nie jak.
git commit -m "Krótki opis zmian"

# Przykłady dobrych opisów:
git commit -m "Dodaj generowanie PDF dla klientów"
git commit -m "Napraw błąd kalkulacji czasu zwrotu"
git commit -m "Zmień port serwera z 3000 na 3777"
Wyślij zmiany na GitHub
git push
Pełny cykl — wszystko naraz
git add . && git commit -m "Opis zmian" && git push
9
Po sesji Claude Code — wysyłanie zmian na GitHub
Po tym jak Claude Code wprowadzi zmiany w projekcie, warto je przejrzeć i wysłać do backupu. Oto bezpieczny sposób.
Sprawdź co Claude zmienił
Zawsze warto wiedzieć co dokładnie trafia na GitHub przed wysłaniem.
git status                  # lista zmienionych plików
git diff                    # szczegółowy podgląd zmian linia po linii
git diff --stat             # skrócone podsumowanie: ile linii w każdym pliku
Dodaj wybrane pliki (bezpieczniej niż git add .)
Jeśli Claude zmieniał konkretne pliki, lepiej je wskazać ręcznie niż dodawać wszystko naraz.
git add server.js
git add src/pdf.js src/db.js
git add public/js/views/calculator.js
git add public/css/style.css
Lub dodaj wszystko (szybka opcja, gdy ufasz zmianom)
git add .
# Upewnij się że .gitignore wyklucza .env i data/ !
Commit z opisem co Claude zrobił w tej sesji
git commit -m "Dodaj integrację OpenAI — raport tygodniowy AI"
git commit -m "Napraw generowanie PDF — zmiana na systemowy Chrome"
git commit -m "Dodaj foldery klientów przy tworzeniu kontaktu"
Wyślij backup na GitHub
git push
Uruchamianie komend Git bezpośrednio w sesji Claude Code
W terminalu Claude Code możesz poprzedzić komendę znakiem ! — komenda wykona się w bieżącej sesji i wynik wróci do konwersacji.
# W polu czatu Claude Code wpisz:
! git status
! git add . && git commit -m "Opis zmian" && git push
! git log --oneline -5
ℹ
Możesz też poprosić Claude Code słownie: "Zrób commit z opisem X i wypchnij na GitHub" — Claude uruchomi odpowiednie komendy git za Ciebie.
10
Podgląd historii zmian
Jak sprawdzić co i kiedy zostało zmienione w projekcie.
Historia commitów
git log                      # pełna historia z datami i autorami
git log --oneline            # skrócona — jeden wiersz na commit
git log --oneline -10        # ostatnie 10 commitów
git log --oneline --graph    # z grafem gałęzi
Co zmieniało się w konkretnym pliku
git log --oneline -- server.js
git log -p -- src/db.js         # z podglądem zmian linia po linii
Kto i kiedy zmienił konkretną linię
git blame server.js
11
Cofanie zmian
Jak wróciić do wcześniejszej wersji — przed lub po commitcie.
Cofnij zmiany w konkretnym pliku (niezacommitowane)
git restore server.js        # przywróć jeden plik do ostatniego commita
git restore .                # przywróć WSZYSTKIE pliki — ostrożnie!
Cofnij ostatni commit (zachowaj zmiany w plikach)
Commit znika z historii, ale zmiany w plikach zostają — możesz je poprawić i zcommitować jeszcze raz.
git reset HEAD~1              # cofnij 1 commit wstecz
git reset HEAD~2              # cofnij 2 commity wstecz
Usuń plik ze staging (przed commitem)
git restore --staged plik.js  # cofnij "git add" dla pliku
git restore --staged .        # cofnij "git add" dla wszystkiego
⚠
git reset --hard usuwa zmiany z plików bezpowrotnie. Nie używaj jeśli nie jesteś pewien — użyj git reset bez --hard.
12
Klonowanie projektu na nowy komputer
Jak pobrać projekt z GitHub na nowy Mac lub po reinstalacji systemu.
Sklonuj repozytorium
git clone https://github.com/twoj-login/nazwa-projektu
cd nazwa-projektu
Pobierz najnowsze zmiany z GitHub (gdy repo już masz)
git pull                     # pobierz i scal zmiany z origin/main
Lista Twoich repozytoriów
gh repo list                  # wszystkie Twoje repo na GitHub
gh repo list --limit 20     # z limitem wyników
13
Komendy pomocnicze GitHub CLI (gh)
Przydatne komendy do zarządzania repozytoriami i kontem GitHub z Terminala.
Zarządzanie repozytoriami
gh repo list                              # lista Twoich repo
gh repo view                              # info o aktualnym repo
gh repo view --web                       # otwórz repo w przeglądarce
gh repo create nowa-nazwa --private     # nowe prywatne repo
gh repo create nowa-nazwa --public      # nowe publiczne repo
gh repo delete nowa-nazwa               # usuń repo (potwierdź nazwą)
Zmiana widoczności repo (private/public)
gh repo edit --visibility private       # zmień na prywatne
gh repo edit --visibility public        # zmień na publiczne
Konto i autoryzacja
gh auth status                            # sprawdź czy jesteś zalogowany
gh auth login                             # zaloguj się ponownie
gh auth logout                            # wyloguj się
gh auth refresh -s repo                  # odśwież uprawnienia
Szybki podgląd repo w przeglądarce
gh browse                                 # otwórz główną stronę repo
gh browse server.js                       # otwórz konkretny plik na GitHub
