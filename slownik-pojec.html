1
Terminal
Środowisko · Jak rozmawiać z komputerem tekstem
🖥️
Terminal / wiersz poleceń
Środowisko
Analogia
Zamiast klikać po menu restauracji na tablecie, mówisz kelnerowi wprost: „podwójne espresso, bez cukru". Terminal to właśnie to — piszesz komputerowi co ma zrobić, bez klikania po oknach. Szybciej i precyzyjniej.
Terminal to program w którym wpisujesz tekstowe polecenia. Komputer je wykonuje i pokazuje wynik jako tekst. Na macOS otworzysz go przez Cmd + Spacja i wpisując „Terminal".

Większość pracy z kodem dzieje się właśnie tu — uruchamianie serwera, instalowanie bibliotek, wysyłanie na GitHub.
💡
W Claude Code wpisz ! polecenie — wykrzyknik uruchamia polecenie w terminalu bez wychodzenia z rozmowy z Claude.
W tej aplikacji

Uruchamiasz serwer poleceniem npm start. Zatrzymujesz go przez pkill -f "node server.js".

2
localhost i port
Środowisko · Twoja prywatna sieć
🏠
localhost
Sieć
Analogia
Numer wewnętrzny telefonu w biurze. Możesz zadzwonić na „200" i odebrać w pokoju obok — ale ten numer nie istnieje poza budynkiem. Nikt z zewnątrz nie może się połączyć. localhost działa tak samo — to adres istniejący tylko na Twoim komputerze.
localhost to specjalny adres oznaczający „ten komputer". Gdy wpisujesz http://localhost:3777 w przeglądarce, nie łączysz się z internetem — rozmawiasz z programem działającym na Twoim własnym dysku. Nikt z zewnątrz tego nie widzi.
W tej aplikacji

LED Offer Studio działa pod http://localhost:3777 — tylko na Twoim komputerze, kompletnie offline.

🚪
Port
Sieć
Analogia
Numer mieszkania w bloku. Blok (komputer) ma jeden adres, ale w środku jest wiele mieszkań (programów). Każde ma swój numer i każde „odbiera" tylko połączenia skierowane do niego.
Port to numer (0–65535) który identyfikuje konkretny program nasłuchujący na połączenia. Na jednym komputerze może działać wiele programów jednocześnie — każdy na innym porcie. Jeśli port jest już zajęty przez inny program, nowy nie może się uruchomić.
W tej aplikacji

Serwer działa na porcie 3777. Port 3000 był już zajęty przez inną aplikację — dlatego go zmieniliśmy. Sprawdzisz co zajmuje port: lsof -ti:3777

3
Node.js
Środowisko · Silnik aplikacji
⚙️
Node.js
Środowisko
Analogia
JavaScript to język. Przeglądarka (Chrome, Safari) rozumie ten język, ale tylko w kontekście stron www — jak aktor który gra wyłącznie w teatrze. Node.js to ten sam aktor, ale teraz może grać wszędzie — w kinie, na ulicy, w firmie. Kod może działać bezpośrednio na komputerze, bez przeglądarki.
Node.js to środowisko uruchomieniowe dla JavaScript poza przeglądarką. Dzięki niemu piszesz serwer, skrypty i narzędzia w tym samym języku co interfejs użytkownika. Instalujesz go raz przez Homebrew (brew install node) i używasz we wszystkich projektach.
W tej aplikacji

Cała aplikacja to Node.js. Plik server.js to serce — uruchamiasz go przez npm start.

4
npm i zależności
Środowisko · Gotowe klocki do budowania
📦
npm i zależności (dependencies)
Narzędzie
Analogia
App Store dla programistów. Zamiast pisać od zera kod do generowania PDF albo obsługi AI — pobierasz gotowy, przetestowany przez tysiące ludzi pakiet i go używasz. npm to ten sklep, a zależności to lista aplikacji które zainstalowałeś.
npm (Node Package Manager) to narzędzie do instalowania gotowych bibliotek. Jedno polecenie npm install pobiera wszystko czego projekt potrzebuje.

Pobrane pakiety lądują w folderze node_modules/ — może ważyć setki MB. Dlatego nigdy nie wrzucasz go na GitHub — jest w .gitignore. Każdy kto pobierze projekt, robi npm install i sam pobiera swoje kopie.
W tej aplikacji

Używamy: express (serwer), puppeteer (PDF), openai (AI), dotenv (tajne klucze). Instalujesz wszystko jednym poleceniem: npm install

5
Plik .env
Pliki konfiguracyjne · Tajne ustawienia
🔐
.env
Bezpieczeństwo
Analogia
Kartka z PINami do kart — trzymasz ją w zamkniętej szufladzie, osobno od portfela. Portfel (kod aplikacji) możesz gdzieś zostawić czy pokazać, ale kartka z PINami zostaje w szufladzie tylko u Ciebie.
Plik .env przechowuje tajne dane aplikacji: klucze API, hasła, adresy serwerów. Każda linia to para NAZWA=wartość. Aplikacja czyta je przez process.env.NAZWA.

Ten plik nigdy nie trafia na GitHub — jest w .gitignore. Dzięki temu tajne dane nie są wpisane w kodzie i nie wyciekają przy udostępnianiu projektu. 
# Plik .env — tylko na Twoim komputerze, nigdy na GitHub
OPENAI_API_KEY=sk-proj-twój-tajny-klucz
OPENAI_MODEL=gpt-4o-mini
⚠️
Jeśli klucz API wycieknie na GitHub — ktoś może używać Twojego konta OpenAI na Twój koszt. Jeśli to się stanie: natychmiast usuń klucz w panelu OpenAI i wygeneruj nowy.
6
Plik .gitignore
Pliki konfiguracyjne · Lista rzeczy do pominięcia
🙈
.gitignore
Git
Analogia
Karteczka „NIE PAKOWAĆ" przyklejona na rzeczy przy przeprowadzce. Firma przeprowadzkowa (Git) bierze wszystko — oprócz rzeczy z karteczką. Przyklejasz ją na: tajne pliki, dane klientów, gigantyczne foldery z bibliotekami.
Plik .gitignore to lista plików i folderów, których Git ma nie śledzić i nie wysyłać na GitHub. Każda linia to wzorzec nazwy. Raz skonfigurowany działa automatycznie — nie musisz za każdym razem pamiętać co pominąć. 
# Plik .gitignore — Git ignoruje wszystko co tu wpisano
node_modules/   ← setki MB bibliotek, każdy instaluje sam
.env            ← klucze API i hasła
data/           ← baza danych z danymi klientów
Klienci/        ← dokumenty klientów
.DS_Store       ← systemowe śmieci macOS

package.json
Pliki konfiguracyjne · Dowód tożsamości projektu
📋
package.json
Konfiguracja
Analogia
Etykieta na pudełku z ciastem — mówi jak się nazywa, co jest w środku (składniki = zależności) i jak je podgrzać (skrypty = jak uruchomić). Każdy projekt Node.js ma tę etykietę.
package.json to plik w głównym folderze projektu zawierający: nazwę projektu, wersję, listę wymaganych bibliotek (zależności) i skrypty skrótowe.

Gdy wpisujesz npm start — npm sprawdza w package.json co ma uruchomić. Gdy wpisujesz npm install — pobiera wszystkie biblioteki z listy zależności.
W tej aplikacji

W package.json jest zdefiniowane "start": "node server.js" — stąd npm start uruchamia serwer. Są też wymienione: express, puppeteer, openai, dotenv i inne.

Markdown (.md)
Pliki konfiguracyjne · Format dokumentacji
📝
Markdown
Format
Analogia
Stenografia dla tekstu. Zamiast klikać „Pogrub" w Wordzie, piszesz **słowo** i jest pogrubione. Zamiast wstawiać nagłówek — piszesz # Tytuł. Prosty zwykły tekst, który wygląda ładnie gdy go wyrenderujesz.
Pliki z rozszerzeniem .md to Markdown. GitHub automatycznie renderuje je jako sformatowane strony — dlatego prawie cała dokumentacja open-source jest w tym formacie. Claude Code też odpowiada w Markdown.
W tej aplikacji

Cały folder dokumentacja/ to pliki .md: README.md, API.md, DATABASE.md itd. GitHub wyświetla je jako czytelne strony w repozytorium.

CLAUDE.md
Pliki konfiguracyjne · Pamięć projektu dla Claude
🧠
CLAUDE.md
Claude Code
Analogia
Instrukcja przekazania obowiązków dla nowego pracownika. Każda sesja Claude Code to jak nowy dzień — Claude nie pamięta poprzedniej rozmowy. CLAUDE.md to notatka którą zostawiasz na biurku: „projekt nazywa się tak, serwer działa na tym porcie, używamy tego klucza, ostatnia wersja to v1.1.1".
CLAUDE.md to plik w głównym folderze projektu, który Claude Code czyta automatycznie przy każdym otwarciu nowej sesji. Dzięki niemu Claude od razu wie co to za projekt, jak go uruchomić, co jest zrobione i jak pracujemy.
✅
Po każdej sesji w której coś zmieniono — zaktualizuj CLAUDE.md. To 2 minuty pracy, które oszczędzają 20 minut tłumaczenia na początku każdej kolejnej sesji.
W tej aplikacji

Plik CLAUDE.md w głównym folderze zawiera: aktualną wersję, port, tech stack, mapę plików, co jest wdrożone i zasady pracy z GitHubem.

10
Git
Git i GitHub · Nieograniczone „cofnij"
🕐
Git — kontrola wersji
Git
Analogia
Przycisk „Cofnij" w Wordzie, ale bez limitu kroków i z opisem każdej zmiany. Możesz cofnąć się do dokładnie tego momentu sprzed 3 tygodni kiedy wszystko działało — i zobaczyć co dokładnie wtedy było w każdym pliku.
Git to narzędzie śledzące historię zmian w plikach projektu. Działa lokalnie na Twoim komputerze. Sam decydujesz kiedy „zapisać" stan (commit) i kiedy wysłać na serwer (push). Daje poczucie bezpieczeństwa — zawsze możesz wrócić do poprzedniej działającej wersji.
11
Repozytorium
Git i GitHub · Folder z pamięcią
🗂️
Repozytorium (repo)
Git
Analogia
Zwykły folder na projekty budowlane + pełne archiwum zdjęć z każdego etapu budowy. Nie tylko widzisz aktualny stan domu — widzisz jak wyglądał po każdym tygodniu pracy, kto co zmienił i dlaczego.
Repozytorium to folder projektu, w którym Git śledzi zmiany. Tworzy się je poleceniem git init. Git tworzy wtedy ukryty folder .git/ — tam przechowuje całą historię. Repozytorium może być lokalne (tylko na Twoim dysku) lub zdalne (np. na GitHubie, jako backup).
12
Commit
Git i GitHub · Zapisana migawka z opisem
📸
Commit
Git
Analogia
Kliknięcie „Zapisz" — ale do każdego zapisu dołączasz kartkę z notatką „co zmieniłem i dlaczego". Każda kartka ma datę i autora. Pół roku później możesz przejrzeć całą historię zapisów i znaleźć dokładnie ten w którym coś poszło nie tak.
Commit to zapisana migawka stanu wszystkich plików projektu w danym momencie. Każdy commit ma: unikalny ID, datę, autora i wiadomość opisującą co zrobiono.

Zanim zapiszesz, dodajesz pliki do „poczekalni" (git add), potem zatwierdzasz (git commit -m "opis"). 
# Dodaj zmienione pliki do poczekalni
git add server.js

# Zapisz migawkę z opisem co i dlaczego
git commit -m "Napraw generowanie PDF — zmiana na system Chrome"
13
Branch (gałąź)
Git i GitHub · Eksperymentuj bez ryzyka
🌿
Branch
Git
Analogia
Notatnik z kartkami do wyrwania. Chcesz przetestować nowy pomysł? Piszesz go na osobnej kartce — nie w środku notatnika. Pomysł wyszedł? Wklejasz kartkę. Nie wyszedł? Wyrzucasz bez śladu.
Branch to odgałęzienie historii projektu — możesz pracować nad nową funkcją na osobnej gałęzi, nie ryzykując zepsucia działającej wersji. Główna gałąź to zazwyczaj main. W małych projektach z jednym deweloperem często pracuje się wyłącznie na main.
14
Push i Pull
Git i GitHub · Synchronizacja z chmurą
↕️
git push / git pull
Git
Analogia
Synchronizacja zdjęć z iPhone na iCloud. Push = wyślij zdjęcia z telefonu do chmury. Pull = pobierz zdjęcia z chmury na nowy telefon. Twój komputer i GitHub to dwa urządzenia z tą samą „biblioteką" — musisz je synchronizować.
git push wysyła Twoje lokalne commity na GitHub — aktualizuje backup i udostępnia zmiany.
git pull pobiera z GitHub zmiany których nie masz lokalnie — np. gdy pracujesz na dwóch komputerach.

Ważne: push wysyła tylko commity. Najpierw musisz zrobić commit, potem push.
15
GitHub
Git i GitHub · iCloud dla kodu
🐙
GitHub
Platforma
Analogia
Git to narzędzie (jak aparat fotograficzny), GitHub to miejsce do przechowywania i przeglądania zdjęć (jak iCloud Zdjęcia, ale dla kodu). Możesz mieć Git bez GitHuba — ale GitHub bez Gita nie ma sensu.
GitHub to serwis przechowujący repozytoria git w chmurze. Daje: bezpieczny backup kodu, dostęp z każdego komputera, historię zmian w przeglądarce, możliwość udostępniania. Repozytorium może być publiczne lub prywatne.
W tej aplikacji

LED Offer Studio ma prywatne repo na GitHub. Backup wysyłasz poleceniem git push — ale tylko kiedy Ty zdecydujesz, nie automatycznie.

16
Semantic Versioning
Wersjonowanie · Numer wersji który coś znaczy
🔢
Semantic Versioning — MAJOR.MINOR.PATCH
Wersjonowanie
Analogia
Numer wersji telefonu: iPhone 15.2.1. Pierwsza cyfra = nowa generacja (duże zmiany). Druga = nowa funkcja (aktualizacja systemu). Trzecia = poprawka błędu (łatka bezpieczeństwa). Numer wersji jednym rzutem oka mówi jak duża była zmiana.
Semantic Versioning (semver) to standard numerowania wersji w formacie MAJOR.MINOR.PATCH:

PATCH (np. 1.1.0 → 1.1.1) — poprawka błędu, zmiana konfiguracji
MINOR (np. 1.0.0 → 1.1.0) — nowa funkcja, nic nie zepsute
MAJOR (np. 1.0.0 → 2.0.0) — przepisanie aplikacji, duże zmiany
W tej aplikacji

Startowaliśmy od v1.0.0. Po dodaniu OpenAI, folderów klientów i naprawie PDF — przeszliśmy na v1.1.0 (nowe funkcje). Po dodaniu CLAUDE.md i CHANGELOG.md — v1.1.1 (drobna zmiana).

17
Tag git
Wersjonowanie · Zakładka w historii
🏷️
Tag git
Git
Analogia
Zakładka z podpisem wklejona w konkretne miejsce w książce. Zamiast pamiętać że „wersja 1.1.0 to ten commit z 26 czerwca" — przyklejasz zakładkę z napisem „v1.1.0" i możesz do niej wrócić w każdej chwili.
Tag to nazwana etykieta przypisana do konkretnego commita w historii Gita. Używasz tagów do oznaczania wersji. Tagi wysyłasz na GitHub osobnym poleceniem. 
# Utwórz tag dla aktualnego commita
git tag -a v1.2.0 -m "Dodaj nową funkcję X"

# Wyślij tagi na GitHub (osobno od git push)
git push --tags

CHANGELOG.md

📓
CHANGELOG.md
Wersjonowanie
Analogia
Dziennik budowy domu. „14 maja: wylano fundamenty. 28 maja: postawiono ściany. 10 czerwca: zamontowano okna." Za rok wiesz co, kiedy i dlaczego zostało zmienione — bez grzebania w setkach commitów.

API
Komunikacja · Jak programy rozmawiają ze sobą
🔌
API — Application Programming Interface
Komunikacja
Analogia
Menu w restauracji. Nie wchodzisz do kuchni i nie gotujesz sam. Patrzysz w menu (API), zamawiasz „raport z tego tygodnia", kuchnia (serwer OpenAI) przygotowuje i przynosi gotowe. Nie musisz wiedzieć jak działa kuchnia — tylko co możesz zamówić i jak.
API to zestaw zasad określający jak jeden program może komunikować się z drugim. Twoja aplikacja wysyła zapytanie do API OpenAI (zamówienie), OpenAI przetwarza je i odsyła odpowiedź (danie). API może być wewnętrzne (frontend ↔ backend tej samej aplikacji) lub zewnętrzne (do obcych serwisów).
W tej aplikacji

LED Offer Studio ma własne API (GET /api/clients = pobierz klientów) i używa zewnętrznego API OpenAI do generowania raportów AI.

20
Klucz API
Komunikacja · Karta członkowska do zewnętrznych usług
🗝️
API Key — klucz API
Bezpieczeństwo
Analogia
Karta członkowska klubu. Okazujesz ją przy każdym wejściu — klub wie kto wchodzi i nalicza opłaty na właściwe konto. Jeśli zgubisz kartę i ktoś ją znajdzie — wchodzi zamiast Ciebie i rachunek płacisz Ty.
Klucz API to długi, unikalny ciąg znaków który dołączasz do każdego zapytania do zewnętrznej usługi. Usługa wie dzięki niemu kto wysyła zapytanie i czy ma prawo to robić. Klucze są tajne — trzymasz je wyłącznie w pliku .env.
⚠️
Nigdy nie wklejaj klucza API w kod. Nigdy nie wysyłaj go na GitHub. Jeśli wycieknie — natychmiast go unieważnij w panelu usługi i wygeneruj nowy.
21
Endpoint
Komunikacja · Numer telefonu do konkretnego działu
📍
Endpoint / REST
Komunikacja
Analogia
Numer telefonu do konkretnego działu firmy. /api/clients to „dział obsługi klientów", /api/pdf to „dział dokumentów". Dzwonisz pod konkretny numer (endpoint) i dostajesz konkretną usługę. Metoda HTTP (GET/POST) to jak kierunek — „chcę pobrać" albo „chcę dodać".
Endpoint to konkretny adres URL + metoda HTTP, pod którym można wykonać akcję:

GET — pobierz dane  ·  POST — dodaj nowe  ·  PUT/PATCH — zmień istniejące  ·  DELETE — usuń 
# Przykłady endpointów w tej aplikacji:
GET  /api/clients     ← lista klientów
POST /api/clients     ← dodaj klienta
GET  /api/clients/5   ← konkretny klient
POST /api/ai/report   ← generuj raport AI

JSON
Komunikacja · Wspólny język danych
📄
JSON
Format danych
Analogia
Formularz z polami. Zamiast pisać „Jan ma 35 lat i mieszka w Krakowie" (trudno przetworzyć automatycznie), wpisujesz dane w pola: Imię: Jan, Wiek: 35, Miasto: Kraków. Każdy program na świecie umie taki formularz odczytać i zrozumieć.
JSON to format tekstowy do przesyłania danych między programami. Pary klucz-wartość w nawiasach klamrowych. To de facto standard komunikacji przez API. 
// Przykładowy JSON — dane klienta

}
23
HTTP i HTTPS
Komunikacja · Protokół internetu
🌐
HTTP / HTTPS
Protokół
Analogia
HTTP to kartka pocztowa — każdy kto ją widzi po drodze może ją przeczytać. HTTPS to list w zaklejonej kopercie — zaszyfrowany, tylko adresat otwiera. Ta sama wiadomość, ale zupełnie inna ochrona.
HTTP to protokół komunikacji między przeglądarką a serwerem — zasady wymiany danych w internecie. HTTPS to szyfrowana wersja (S = Secure). Lokalna aplikacja działa na http:// — szyfrowanie nie jest potrzebne gdy rozmawiasz sam ze sobą. Publiczne aplikacje w internecie używają zawsze https://.
24
Frontend i Backend
Architektura · Witryna i kuchnia
🏗️
Frontend vs Backend
Architektura
Analogia
Sklep. Frontend = witryna sklepowa, ekspozycja, kasa — to co widzi klient i z czym wchodzi w interakcję. Backend = magazyn, biuro rachunkowe, system zamówień — niewidoczne dla klienta, ale tam dzieje się prawdziwa praca.
Frontend to kod działający w przeglądarce — wszystko co widzisz i klikasz. Pliki HTML, CSS, JavaScript.

Backend to kod działający na serwerze — przetwarza dane, zapisuje do bazy, generuje PDF, komunikuje się z OpenAI. Użytkownik go nie widzi.
W tej aplikacji

Frontend: folder public/ — HTML, CSS, JS działający w Twojej przeglądarce.
Backend: server.js — Node.js który odpowiada na zapytania, zarządza bazą i generuje PDFy.

// Tak aplikacja czyta tajne dane (server.js):
const apiKey = process.env.OPENAI_API_KEY;
// Wartość pochodzi z .env — nigdy z kodu wprost

Wysyłanie projektu na GitHub
Co sprawdzić zanim klikniesz „wyślij"
Sprawdź co Git widzi: git status Uruchom git status i przejrzyj listę plików gotowych do commita. Czy są tam rzeczy które nie powinny trafić na GitHub?
Upewnij się że .env i dane klientów są w .gitignore Jeśli widzisz .env lub data/ w git status jako „do dodania" — zatrzymaj się i dodaj je do .gitignore zanim cokolwiek wyślesz.
Napisz sensowny opis commita „update" to zły opis. „Dodaj integrację z OpenAI — raport tygodniowy" to dobry. Za rok będziesz wdzięczny sobie za konkretne opisy.
Dodaj tag wersji przy ważniejszych zmianach git tag -a v1.2.0 -m "Dodaj funkcję X" — oznacz commit numerem wersji. Tagi wysyłasz osobno: git push --tags
Zaktualizuj CHANGELOG.md przed wysłaniem Dopisz w CHANGELOG.md co zmieniło się w nowej wersji. To 5 minut które sprawią że za rok będziesz wiedział co, kiedy i dlaczego.
